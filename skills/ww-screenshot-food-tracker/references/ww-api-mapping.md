# WeightWatchers API Mapping (Undocumented)

This skill targets undocumented web endpoints observed in the WeightWatchers web app. Endpoints and payloads may change.

## Recommended execution split

1. Screenshot extraction -> `foods_raw.json` (name, portionSize, mealTime, date)
2. Resolve -> `scripts/ww_resolve_foods.py` -> `foods_resolved.json`
3. Track -> `scripts/ww_track_resolved.py` -> POST writes + optional summary check

## Authentication flow

1. `POST https://auth.weightwatchers.<tld>/login-apis/v1/authenticate`
   Request body:

```json
{
  "username": "<email>",
  "password": "<password>",
  "rememberMe": false,
  "usernameEncoded": false,
  "retry": false
}
```

2. Read `tokenId` from the response.
3. `GET https://auth.weightwatchers.<tld>/openam/oauth2/authorize?response_type=id_token&client_id=webCMX&redirect_uri=https%3A%2F%2Fcmx.weightwatchers.<tld>%2Fauth&nonce=<random>`
   with header `Cookie: wwAuth2=<tokenId>`.
4. Extract `id_token` from redirect fragment (`#id_token=...`).
5. Use `Authorization: Bearer <JWT>` for subsequent API calls.

Supported by `scripts/ww_auth_jwt.py`.

## Domain selection

Set `<tld>` by country:

- Germany: `de`
- United States: `com`

Use matching `WW_API_BASE_URL`, for example `https://cmx.weightwatchers.de`.

## Food search endpoint

Default script path:

- `WW_API_SEARCH_PATH=/api/v3/search/foods`
- query parameter: `WW_API_SEARCH_QUERY_PARAM=query`
- optional extra params: `WW_API_SEARCH_EXTRA_PARAMS` (query-string format)

The script supports several common response containers (`results`, `foods`, `items`, `data`, `hits`) and nested variants.

APK evidence (decompiled):

- `sources/com/weightwatchers/search/retrofit/FoodSearchService.java`

- `sources/com/weightwatchers/food/common/util/FoodsSearchQueryBuilder.java`
  - confirms query-map key names listed above

## Track food endpoint

Default script path:

- `WW_API_TRACK_PATH=/api/v4/cmx/members/~/trackedFoods/{date}`

Default POST payload generated by `scripts/ww_food_tracker.py` (list body):

```json
[
  {
    "_id": "resolved-food-id",
    "versionId": "resolved-version-id",
    "sourceType": "WWFOOD",
    "timeOfDay": "MIDDAY",
    "portionSize": 1.0,
    "portionId": "resolved-portion-id",
    "mealId": null,
    "mealVersionId": null,
    "mealType": null,
    "entryId": null,
    "isQuickAdd": false
  }
]
```

APK evidence (decompiled):

- `sources/com/weightwatchers/food/common/service/FoodTrackingService.java`
  - `@POST("/api/v4/cmx/members/~/trackedFoods/{date}")` with `@Body List<Request>`
  - `@PATCH`/`@DELETE` on `/api/v3/cmx/members/~/trackedFoods/{date}`
- `sources/com/weightwatchers/food/common/requests/TrackItemRequestJsonAdapter.java`
  - JSON field names: `_id`, `versionId`, `sourceType`, `timeOfDay`, `portionSize`, `portionId`, `mealId`, `mealVersionId`, `mealType`, `entryId`, `isQuickAdd`

For live requests, ensure `versionId` and `portionId` come from search hit data (or explicit user input where valid).

## Read-only day summary

Example summary endpoint:

- `GET /api/v4/cmx/operations/composed/members/~/my-day-summary/{YYYY-MM-DD}`

Example:

```bash
curl -H "Authorization: Bearer <JWT>" \
  -H "Accept: application/json" \
  "https://cmx.weightwatchers.de/api/v4/cmx/operations/composed/members/~/my-day-summary/2026-02-20?noNegativeWeeklies=false&useHTS=true"
```

Tracker integration:

- `scripts/ww_food_tracker.py --verify-summary --summary-date YYYY-MM-DD` fetches this endpoint after write calls.
- Override via `WW_API_SUMMARY_PATH_TEMPLATE` and `WW_API_SUMMARY_QUERY` if needed.

## Safety notes

- Treat JWT and credentials as secrets.
- Run `--dry-run` first before creating account mutations.
- Do not run write endpoints unless payload fields and side effects are understood.
